## 🛠️ Why is this file used in our architecture?

When the user sees the processed product on the dashboard (**Frontend**) and clicks the **"Publish"** button, the following process takes place:

1. **Frontend Request:** The Frontend sends a request to the FastAPI backend at the endpoint: `/api/products/publish`.
2. **Backend Processing:** FastAPI receives the marketing copy (Telegram, Facebook, Pinterest texts) previously generated by Groq.
3. **Webhook Trigger:** FastAPI does not publish the content directly. Instead, it triggers a **Webhook** (sends a fast HTTP POST request) to the n8n server running alongside it within the same Docker network.
4. **n8n Pipeline Execution:** This is where the JSON file comes into play. It configures n8n on how to accept the incoming payload, parse it, and dispatch it simultaneously to the user's Telegram channel, Facebook page, or Pinterest account using built-in integrations (**Nodes**).

> 💡 **Note:** n8n operates via a visual workflow interface, but behind the scenes, all node connections and configurations are saved and maintained as a structured JSON file.

---

### 🔍 JSON Structure Breakdown

* **`n8n-nodes-base.webhook`**
    A Webhook node that listens on a local path within the Docker environment. It triggers the entire workflow the moment it receives data from FastAPI.
* **`n8n-nodes-base.telegram`**
    A built-in n8n node that securely loads your bot credentials (`TELEGRAM_BOT_TOKEN`) and dynamically extracts the specific pre-generated text from the incoming FastAPI payload using: `{{ $json.body.aiCopy.telegram }}`.
* **`connections`**
    Defines the virtual wiring. Once the Webhook successfully executes, the data flows automatically and seamlessly directly into the Telegram node (and subsequently to Facebook and Pinterest in parallel).
